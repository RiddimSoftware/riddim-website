The Riddim Software Factory

Riddim ships software through an autonomous delivery loop — this page describes how.

What "software factory" means

The term "software factory" is 57 years old. Hitachi coined it in 1969 to describe a production-line approach to writing code — standardised processes, repeatable outcomes, measurable throughput. It is documented on Wikipedia and revisited in a 2026 essay by Mager: The Software Factory. What is new now is that autonomous agents can play specific, repeatable roles in that production line — writing code, opening pull requests, reviewing diffs, and merging — without a human in the merge path. Riddim is running one such loop in production.

How the loop works

Every change that ships through Riddim follows the same ordered sequence:

  1. Intake. Work is filed as a Jira ticket by the Backlog Team agent. Tickets include acceptance criteria, inputs, risks, and a definition of done before any code is written.
  2. Orchestration. The Sprint Planning and Scope Orchestrator agents pick eligible tickets, assign them to the next available slot, and spawn a developer session for each one.
  3. riddim-developer-bot[bot] opens a pull request. A Claude-powered developer agent checks out the ticket, implements the change in a git worktree, and opens a pull request labeled autonomous. The branch name and PR title carry the Jira ticket key.
  4. riddim-reviewer-bot[bot] reviews and merges. A separate Claude-powered reviewer agent reads the diff and the ticket's acceptance criteria, makes any needed fixup commits, approves, and squash-merges. The two-bot identity contract prevents self-approval: developer-bot opens, reviewer-bot closes.
  5. Jira reconciliation. The orchestrator reads PR state on each cycle and transitions Jira tickets to Done when the corresponding PR is merged. The board stays in sync with the merge log without manual updates.

No human approves or merges the pull request. Humans file issues, set priorities, and review the board. The loop handles the rest.

Built on

Riddim's loop runs on two open infrastructure components it did not build.

Gas City is the agent coordination layer. It manages work queues, agent lifecycles, and inter-agent messaging. Riddim is a production user of Gas City, not its author.

Beads is the issue tracker used internally by the loop. It stores work items, tracks state, and provides the command-line interface agents use to claim and close tasks. Again, Riddim is a user, not the originator of the software factory concept or of Beads.

The claim this page makes is narrower: Riddim is running this loop in production and has the PR history to show for it.
